Mini-X Router advanced SPA router for Mini-X

Lightweight, feature-rich routing for Mini-X reactive framework. Nested layouts, named views, lazy loading, keep-alive, navigation guards, and full reactive route state — all in ~8kb gzipped.

✨ Introduction

Mini-X Router is the official routing solution for Mini-X — a tiny reactive UI library. It offers a Vue Router–like API but deeply integrated with Mini-X reactivity, directives, and component lifecycle. Perfect for SPAs, dashboards, and advanced demos.

✅ Works with mini-x.js core. Provides x-router-view, x-link, x-route directives, programmatic navigation, and route meta fields.

📦 Installation

Include mini-x.js (core) and plugin-router.js in your HTML:

<script src="./mini-x.js"></script>
<script src="./plugin-router.js"></script>

Then create a router instance and install it into your Mini-X app.

⚡ Basic Setup

const routes = [
  { path: '/', component: 'HomePage', name: 'home' },
  { path: '/about', component: 'AboutPage', name: 'about' }
];

const router = MiniXRouter.createRouter({
  history: MiniXRouter.createWebHistory('/'),
  routes,
  debug: true
});

MiniX.createApp(AppShell)
  .component('HomePage', HomePage)
  .component('AboutPage', AboutPage)
  .use(router)
  .mount('#app');

Inside your template, use <div x-router-view></div> as outlet.

🧩 Route Definition

Routes support paths, names, nested children, named views, redirects, meta, and props.

{
  path: '/workspace',
  components: { default: 'WorkspaceLayout', sidebar: 'SidebarComp' },
  meta: { requiresAuth: true, keepAlive: true },
  beforeEnter: (to, from) => { /* custom guard */ },
  children: [
    { path: '', component: 'WorkspaceOverview', name: 'workspace' },
    { path: 'projects', component: 'WorkspaceProjects' }
  ]
},
{
  path: '/users/:id',
  component: 'UserProfile',
  props: true, // pass params as props
  name: 'userDetails'
}

Route meta fields

meta.title (string or function) sets document title automatically. meta.keepAlive enables component caching. Custom meta like requiresAuth can be used in guards.

🖱️ Navigation & Directives

Programmatic navigation

// inside component context (this.$router)
this.$router.push('/analytics')
this.$router.push({ name: 'userDetails', params: { id: 5 }, query: { tab: 'billing' } })
this.$router.replace('/home')
this.$router.back()

Directives: x-link & x-route

x-link accepts a path string, x-route accepts route name or object literal. Both provide automatic active class and handle SPA navigation.

<a x-link="/dashboard" x-active="active" x-active-mode="startsWith">Dashboard</a>
<a x-route="{ name: 'settingsProfile', params: { id: user.id } }" x-replace>Profile</a>
  • x-active – CSS class when route matches (default "active")
  • x-active-mode – exact or startsWith
  • x-replace – use replace instead of push

🖼️ Router View & KeepAlive

<div x-router-view></div> renders matched component. Supports named views via attribute: x-router-view="sidebar". Depth is automatically detected for nested routes.

Keep-alive : when a route record has meta: { keepAlive: true }, the component instance is cached and reused on revisit. You can clear cache with router.clearKeepAlive().

<div x-router-view></div>          <!-- main outlet -->
<div x-router-view="sidebar"></div> <!-- named view -->

🛡️ Guards & Hooks

Global guards, per-route guards, and view transition hooks. Guards can return true, false, a string path, or a location object to cancel/redirect.

// Global beforeEach
router.beforeEach((to, from) => {
  if (to.meta.requiresAuth && !store.isLoggedIn) return { name: 'login' };
  return true;
});

// Per-route
beforeEnter: (to, from) => { if(to.params.id === 'admin') return '/forbidden'; }

// View transition hooks (animations)
router.beforeRouteLeave(async ({ el }) => {
  el.classList.add('fade-out');
  await delay(150);
});
router.afterRouteEnter(({ el }) => el.classList.add('fade-in'));

⏳ Lazy Loading

Mini-X Router provides a powerful, integrated lazy‑loading system that works without any build tools. You can load components on demand by simply placing them in separate .js files and letting the router handle the injection.

Built‑in Loader (MiniXRouter.Loader)

The router includes a MiniX_Loader class that dynamically injects <script> tags, supports retries, timeouts, and caching. You can either pass a loader instance or a configuration object directly to createRouter.

// Option 1: Pass loader config (router creates internal loader)
const router = MiniXRouter.createRouter({
  history: createWebHistory(),
  loader: {
    baseDir: '/components',   // directory where .js files live
    ext: '.js',               // file extension
    retries: 1,               // number of retries on failure
    timeout: 5000             // load timeout in ms
  },
  routes: [
    {
      path: '/customer',
      component: (load) => load('Customer')   // load function is provided
    }
  ]
});

// Option 2: Use a custom loader instance
const myLoader = new MiniXRouter.Loader('/lazy', { retries: 2 });
const router = MiniXRouter.createRouter({
  loader: myLoader,
  routes: [ /* ... */ ]
});

How It Works

  1. When a route with component: (load) => load('Customer') is activated for the first time, the router calls the function and passes a load(name) function.
  2. The load function uses the configured loader to fetch /components/Customer.js (or the specified baseDir + name + ext).
  3. The loader injects a <script> tag, waits for the file to load, and resolves the promise.
  4. The component file must register itself with MiniX_Component.register('Customer', CustomerClass).
  5. The router then resolves to the registered component and renders it.

The loader caches successful loads, so subsequent visits to the same route will reuse the component without a second network request.

Example: Lazy Component File (Customer.js)

class Customer {
  data() { return { name: '' }; }
  view = `<input x-model="name">`; }
}
// Required registration
if (typeof MiniX_Component !== 'undefined') {
  MiniX_Component.register('Customer', Customer);
}

Preloading & Cache Management

You can preload components during idle time to reduce latency:

const loader = router.getLoader(); // returns the internal loader instance
loader.preload('Customer');       // fetches but doesn't execute (browser cache)

Clear cached components with loader.clearCache('Customer') or loader.clearCache() to force re‑fetch.

📌 Legacy async components (functions returning Promise<Component>) are still fully supported. The new loader pattern is recommended for zero‑build, file‑based lazy loading.

🔧 API Reference

Method / PropertyDescription
createRouter(options)Creates router instance. Options: routes, history, debug, loader (config or Loader instance).
createWebHistory(base)HTML5 history mode (pushState).
createWebHashHistory(base)Hash mode (fallback for file://).
router.push(to) / replace(to)Navigate programmatically.
router.resolve(target)Resolves location to normalized route object.
router.href(target)Generates full URL for given route.
router.beforeEach(fn)Global navigation guard.
router.beforeRouteLeave(fn)Called before a view is unmounted (useful for exit animations).
router.afterRouteEnter(fn)Called after view is mounted/restored.
router.clearKeepAlive(key?)Clear all or specific keep-alive cache.
router.getLoader()Returns the internal MiniX_Loader instance (if configured).
router.onDebug(listener)Subscribe to internal debug events.
router.currentRouteReactive route object with path, params, query, meta, fullPath.

Component instance APIs

Every component mounted by the router gets this.$router, this.$route, this.$params, this.$query. Additionally, reactive getters route, params, query are available directly in templates.

<template>
  <div>Current user ID: {{ params.id }} </div>
  <div>Full path: {{ route.fullPath }} </div>
  <button @click="$router.push('/')">Home</button>
</template>

🐞 Debugging & Events

Enable debug mode: router.enableDebug() or set debug: true in options. Debug events are logged to console and can be captured with router.onDebug(fn).

Events: navigation:start, navigation:finish, lazy:load:start, keepalive:hit, route:resolved, etc.

📌 File protocol support: Router auto-detects file:// and falls back to hash history. Perfect for local demos without a dev server.

📁 Complete Example (with lazy loader)

// router setup
const router = MiniXRouter.createRouter({
  history: MiniXRouter.createWebHashHistory(),
  loader: { baseDir: './components', ext: '.js', retries: 1 },
  routes: [
    { path: '/', component: 'HomePage' },
    { path: '/customer', component: (load) => load('Customer') }
  ]
});

// Customer.js (in ./components/Customer.js)
class Customer { /* ... */ }
MiniX_Component.register('Customer', Customer);

🔄 Nested router-view behavior

The router automatically resolves depth based on parent x-router-view containers. Nested routes render inside the nearest matching router-view without extra configuration.

🎨 Advanced: named views + multiple outlets

components: {
  default: 'MainContent',
  sidebar: 'SidebarPanel',
  footer: 'FooterNote'
}

In your layout: <div x-router-view></div>, <div x-router-view="sidebar"></div>, etc.

⚡ Performance tip: Use meta.keepAlive for heavy dashboards. Manage cache with router.clearKeepAlive() when needed.
