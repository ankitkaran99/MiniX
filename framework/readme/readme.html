MiniX · Deep Documentation
Every API, every directive, every optimization

MiniX is a lightweight, production‑ready reactive framework with a Vue/Alpine-inspired developer experience but a completely independent reactivity engine (MiniX_State + MiniX_Effect). It provides a powerful DOM compiler, class‑based components with lifecycle, scoped state (x-data), event delegation, and a full HTTP client (MiniX_Request). This document covers every internal API, directive modifier, component hook, and performance strategy.

🔁 Core Reactivity: MiniX_State & Effect Tracking

MiniX_State creates deeply reactive proxies that intercept get and set operations, building dependency graphs automatically. It supports nested objects, arrays, Map, Set, and markRaw for escape hatches. Changes are batched via queueMicrotask and notify effects and watchers.

const state = new MiniX_State({ user: { name: "Alice" }, todos: [] });
state.watch("user.name", (newVal) => console.log(newVal));
state.set("user.name", "Bob");
state.push("todos", "learn MiniX");
console.log(state.get("todos.0")); // "learn MiniX"

⚡ MiniX_Effect — fine‑grained scheduler

Effects track every reactive access and re-run when dependencies change. Features: lazy execution, priority, flush timing (pre, post, frame), and batch depth.

const effect = new MiniX_Effect(() => {
  console.log(state.get("counter"));
}, { flush: "post", lazy: true });
effect.run(); // initial run
// automatically re‑runs when counter changes
OptionDefaultDescription
lazyfalseIf true, effect does not run immediately.
schedulernullCustom scheduling function.
flush'pre''pre' (microtask before DOM), 'post' (after DOM), 'frame' (animation frame).
priority0Higher priority runs first within same flush queue.

Batch API: MiniX_Effect._beginBatch() / _endBatch() (exposed via state.batch(fn)). Prevents intermediate updates.

Event modifiers & key filters

@click.prevent, @keyup.enter, @click.stop, .self, .once, .capture. Key filters include enter, escape, tab, space, arrow-up, etc.

<input @keyup.enter="search">

🧩 Component System (Class‑based)

Components are ES6 classes with built‑in reactivity, props, slots, and lifecycle. Register globally via MiniX_Component.register(name, Class) or locally via registerComponents().

class Counter {
  data() { return { count: 0 }; }
  methods = { inc() { this.count++; } };
  computed = { double() { return this.count * 2; } };
  view = `<button @click="inc">{{ count }} / {{ double }}</button>`;
}
MiniX.createApp(Counter).mount("#app");

🎨 View & Layout — New Template System

MiniX now supports view and layout properties for cleaner separation of component content and wrapper shells.

class MyComponent {
  // ✅ Preferred — the component's own content
  view = '<h2>Hello</h2>'

  // ✅ Preferred — the wrapper shell
  layout = MainLayout          // a class, string, or function
}

Why "view"? It follows the Model-View pattern and avoids the ambiguity of "template" (which can mean layout or content).

1 — Basic layout (default yield) — MOST COMMON

// layouts/MainLayout.js
class MainLayout {
  view = `
    <div class="main-layout">
      <nav>My App</nav>
      <main>
        <template x-yield></template>   <!-- view content goes here -->
      </main>
    </div>
  `
}

// pages/Home.js
class Home {
  layout = MainLayout
  view   = '<h2>Welcome home!</h2>'
}

// Renders:
// <div class="main-layout">
//   <nav>My App</nav>
//   <main><h2>Welcome home!</h2></main>
// </div>

2 — Named yield slots — ADVANCED

class AdminLayout {
  view = `
    <div class="admin">
      <aside>
        <template x-yield="sidebar"></template>
      </aside>
      <section>
        <template x-yield></template>    <!-- default slot -->
      </section>
    </div>
  `
}

class Dashboard {
  layout = AdminLayout

  // Named sections are declared inline inside the view using <template x-section="...">
  view = `
    <template x-section="sidebar">
      <a href="/users">Users</a>
      <a href="/settings">Settings</a>
    </template>

    <h1>Dashboard</h1>
    <p>Everything looks good.</p>
  `
  // Anything NOT inside an x-section becomes the default yield content.
}

3 — Sections as a plain object

class Profile {
  layout = AdminLayout

  view = '<p>Profile page content</p>'

  // Named sections as a data property — merged with any inline <template x-section> tags
  sections = {
    sidebar: '<a href="/profile">Profile</a>'
  }
}

4 — Inline layout string

const cardShell = `
  <div class="card">
    <div class="card-body">
      <template x-yield></template>
    </div>
  </div>
`

class ConfirmDialog {
  layout = cardShell
  view   = '<p>Are you sure?</p>'
}

5 — Layout as a function (receives props)

class Wizard {
  layout = (props) => `
    <div class="wizard wizard--step-${props.step}">
      <header>Step ${props.step} of ${props.total}</header>
      <template x-yield></template>
    </div>
  `
  view = '<p>Fill in the form below.</p>'
}

6 — Runtime layout / view switching

class App {
  layout = MainLayout
  view   = '<p>Loading…</p>'

  mounted() {
    // Swap the layout shell at any time — triggers a re-render
    this.$layout(MinimalLayout)

    // Swap just the content — triggers a re-render
    this.$view('<p>Content loaded!</p>')

    // Remove the layout entirely
    this.$layout(null)
  }
}

7 — Multiple pages, one layout

const mainTemplate = `
  <div class='main-layout'>
    <template x-yield></template>
  </div>
`

class Home {
  layout = mainTemplate
  view   = '<h2>Hello</h2>'
}

class Login {
  layout = mainTemplate
  view   = '<h2>Login</h2>'
}

// Home renders:   <div class='main-layout'><h2>Hello</h2></div>
// Login renders:  <div class='main-layout'><h2>Login</h2></div>

Quick reference — yield markers

<template x-yield></template>           Default slot — receives the whole view
                                          (minus any named sections extracted from it)

<template x-yield="sidebar"></template>  Named slot — receives the matching section

<template x-section="sidebar">…</template>  Inline section declaration inside a view
                                               (stripped from default content automatically)

🌍 Global State & Scope Injection

MiniX templates evaluate inside a sandboxed scope (prototype chain ends at null). To share reactive data across components, use app.addScope() to inject variables into every component’s template scope.

const globalStore = new MiniX_State({ theme: "dark" });
const app = MiniX.createApp(Root)
  .addScope(() => ({ store: globalStore.raw() }))
  .mount("#app");

// In any template: {{ store.theme }}
// Update: globalStore.set("theme", "light")

To add methods directly to component instances (this.$store.get()), use app.addInstanceAPI(factory).

⚠️ Always pass globalStore.raw() (the reactive proxy) – not the raw internal object – to preserve reactivity.

⚙️ Compiler & Templating Deep Dive

MiniX_Compiler walks the DOM, caches directives, and generates reactive effects. It supports fast‑path x-for for simple loops (single element, static attributes, one text binding) using VNode diffing and keyed moves. Complex loops use a blueprint system to avoid re‑compilation.

  • Fast‑path detection: _isSimpleLoopTemplate() checks for no nested directives, single text binding.
  • Blueprint loops: _createBlueprintLoopBlock precomputes bindings and updates via bitmask dependency tracking.
  • Interpolation hoisting: Text nodes with {{ }} are hoisted and updated via a single effect per component.
  • Event delegation: click, input, change events are delegated to root for better performance.

Pipes (modifiers) are supported: {{ value | trim | number }}. Built‑in: trim, number, lower, upper, capitalize, json, boolean. Custom via compiler.modifier(name, handler).

🌐 MiniX_Request – Full HTTP Client

Fluent HTTP client with interceptors, retries, caching, progress events, and parallel helpers. Available globally and injected into components as $fetch.

const api = new MiniX_Request('https://api.example.com', { timeout: 5000 });
api.get('/users').json().then(users => ...);
api.post('/item', { name: "test" }).retry(2).cache(5000).json();

// Component usage:
this.$fetch('/profile').then(data => this.user = data);
this.$fetch('/post', { method: 'POST', body: { title: "Hello" } });

Key features

  • Builder methods: .header(), .query(), .body(), .retry(times, delay, factor), .cache(ms).
  • Interceptors: addRequestInterceptor, addResponseInterceptor, addErrorInterceptor.
  • Progress: .onUploadProgress() and .onDownloadProgress() (XHR fallback).
  • Static helpers: MiniX_Request.all(), allSettled(), waterfall(), pool(requests, limit).
  • Abort: .abort(), abortAll().
  • Cache invalidation: invalidate(url, params), clearCache().

🔌 Additional Core APIs

MiniX_Event_Bus – global pub/sub: on(name, cb), once(), emit(name, payload), wildcard '*' support.
MiniX_Provider – hierarchical DI: provide(key, value), inject(key, fallback), createChild().
MiniX_Sanitizer – HTML sanitization (uses DOMPurify if available, else fallback). Configurable allowed tags/attributes.
MiniX_Listener – used internally for $watch, $computed, event binding, auto cleanup.

🧩 Plugins & Extensibility

Plugins can extend the framework by adding global methods, directives, or instance APIs. Define a plugin with install(app) and use MiniX.use(plugin).

const loggerPlugin = {
  install(app) {
    app.addInstanceAPI((component, instance) => ({
      $log: (msg) => console.log(`[${component.ComponentClass.name}]`, msg)
    }));
  }
};
MiniX.createApp(App).use(loggerPlugin).mount("#app");

🛡️ Security Model

All template expressions ({{ }}, directive values) are evaluated inside a with(__scope__) where __scope__ has a null prototype. This prevents access to window, document, eval, or any global object. HTML is sanitized via MiniX_Sanitizer before insertion (x-html).

⚡ Performance Optimizations

  • Effect batching: All state mutations inside state.batch() trigger one flush.
  • Fast‑path x-for: Simple loops use VNode diffing (keyed) without compiling directives per item.
  • Blueprint loops: Complex loops pre‑analyze directive dependencies and update via bitmask.
  • Event delegation: Global listeners for high‑frequency events (click, input).
  • WeakMap caching: Proxy path maps, directive signatures, and template meta are cached.
  • Microtask queue: Updates are scheduled via queueMicrotask to avoid cascading sync reflows.

📘 Advanced Examples

1. Global store with addScope

const store = new MiniX_State({ count: 0 });
const app = MiniX.createApp({
  view: `<button @click="store.set('count', store.get('count')+1)">{{ store.count }}</button>`
}).addScope(() => ({ store: store.raw() })).mount("#app");

2. Scoped x-data with nested component

<div x-data="{ open: false }">
  <button @click="open = !open">Toggle</button>
  <div x-show="open" x-transition.duration.300>Content</div>
</div>

3. Using $fetch with retry and loading state

class UserProfile {
  data() { return { user: null, loading: false }; }
  async mounted() {
    this.loading = true;
    this.user = await this.$fetch('/api/user', { retry: 2 });
    this.loading = false;
  }
  view = `<div><div x-show="loading">Loading...</div><pre x-text="JSON.stringify(user, null, 2)"></pre></div>`; }
}

4. Custom directive: x-click-outside

MiniX.directive('x-click-outside', ({ el, expression, component }) => {
  const handler = (e) => {
    if (!el.contains(e.target)) component.compiler._evaluate(expression, component._createRenderScope());
  };
  document.addEventListener('click', handler);
  return () => document.removeEventListener('click', handler);
});
💡 Tip: Use state.batch() when updating multiple paths to reduce rendering overhead. For extremely large lists, provide stable :key in x-for for optimal DOM patching.
