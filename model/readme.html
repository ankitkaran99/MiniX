Model + Collection Documentation

Documentation for the combined usage of the Model and Collection classes. This setup gives you an Eloquent-style API in JavaScript while still binding cleanly to standalone data, MiniX_State, or store-backed state.

Goal: keep the nice flow like user.name, user.profile.city, and user.roles.pushModel(...) without making the store cry in a corner.

Overview

The two classes are designed to work together:

Model

Represents a single object. It exposes data with property access, supports defaults, relations, serialization, and binding to reactive state paths.

Collection

Represents an array of model items. It supports helpers like findById, pushModel, resetModels, removeById, and more.

Reactive Source

Both classes can bind to MiniX_State-like sources so updates write through the same path instead of mutating disconnected local copies.

Requirements

  • MiniX_State must exist if you want standalone reactive models.
  • For state/store binding, the source should expose: get, set, has, delete, and optionally watch.
  • Collection should be loaded after or alongside Model in environments where you use relations returning collections.

Quick Start

class UserProfile extends Model {}
class Role extends Model {}

class User extends Model {
    defaults() {
        return {
            name: '',
            profile: null,
            roles: []
        };
    }

    relations() {
        return {
            profile: this.hasOne(UserProfile),
            roles: this.hasMany(Role)
        };
    }

    getId() {
        return this.id;
    }
}

const user = new User({
    id: 1,
    name: 'Ankit',
    profile: { city: 'Jaipur' },
    roles: [
        { id: 1, name: 'Admin' },
        { id: 2, name: 'Editor' }
    ]
});

user.name = 'Ankit Karan';
user.profile.city = 'Kota';
user.roles.pushModel({ id: 3, name: 'Viewer' });
user.roles.findById(2).name = 'Content Editor';

Model Class

Model is a proxy-backed wrapper around object-shaped data. It can create its own reactive backing state, or bind to an existing state/store path.

Core responsibilities

  • Property-style access: user.name
  • Defaults through defaults()
  • Relations through relations()
  • Serialization through toJSON()
  • Reactive path binding through fromState() and fromStore()

Main instance methods

Method Purpose Notes
get(attr, fallback) Read a field safely. Useful when you want fallback logic instead of direct property access.
peek(attr, fallback) Read directly from the source without relation wrapping. Handy when you want raw values.
set(attr, value) Set a field value. Equivalent to property assignment for data fields.
fill(data) Bulk-assign fields. Batches writes when the state source supports batching.
assign(data) Alias of fill(). Mostly for ergonomic preference.
watch(attr, callback) Watch a bound path. Only works when the source supports watch().
raw() Return the current raw object-shaped data. Throws if the bound state path is not object-shaped.
clone() Create a detached clone. Uses toJSON().
cloneWith(data) Clone with overrides. Useful for immutable-style updates.
toJSON() Serialize model data to plain JSON-safe objects. Recursively serializes nested models and collections.

Static methods

Method Purpose
make(data, options) Create a new model instance.
fromState(state, path) Bind a model to a MiniX_State-like path.
fromStore(store, path) Bind a model using a store object that exposes a state-like API or state manager.
bind(source, path) Auto-bind to state or store depending on the input.

Collection Class

Collection wraps arrays of model data. It can work as a standalone collection or bind directly to a reactive array path inside state.

Core responsibilities

  • Model-aware array helpers
  • Conversion from plain arrays to model collections
  • Reactive write-through when backed by state
  • Convenient helpers for lookup, mutation, and serialization

Main methods

Method Purpose Notes
Collection.from(ModelClass, items) Create a standalone collection from array data. Converts array items to models lazily when accessed.
at(index) Return the model at the given index. Reactive collections create wrappers against index paths.
first(), last() Convenient accessors. Return a model or null.
push(item), unshift(item) Add plain data or model instances. Serializes values before writing.
pushModel(data), unshiftModel(data) Add plain object data and normalize it to model instances. Great for nicer DX.
reset(items) Replace the full collection using plain items. Expects an array.
resetModels(items) Replace using model data or plain object items. Converts each item through the model class.
findBy(field, value) Find an item by a field. Optimized to scan raw data first, then wrap only the match.
findById(id) Shortcut for id lookup. Uses id by default.
removeBy(field, value), removeById(id) Remove matching items. Returns removed wrapped models.
updateAt(index, patch) Patch a model at an index. Patch must be a plain object.
updateBy(field, value, patch) Patch a found model by field. Returns the updated model or null.
toArray() Return all items as models. Good for iteration and display.
toJSON() Serialize collection items. Returns plain array data.

Relations

Define relations inside your model using hasOne() and hasMany().

class User extends Model {
    relations() {
        return {
            profile: this.hasOne(UserProfile),
            roles: this.hasMany(Role)
        };
    }
}

hasOne

Returns a nested model instance or null. This is cached by relation key for stable repeated access.

hasMany

Returns a Collection. Items are wrapped by index path when reactive, so array structure changes matter.

Binding to State and Store

Bind to state

const user = User.fromState(state, 'user');
user.name = 'Updated';
user.profile.city = 'Jaipur';
user.roles.pushModel({ id: 3, name: 'Viewer' });

Bind to store

const user = User.fromStore(store, 'user');
user.name = 'Updated through store binding';

Store binding works when the store exposes a state-like object directly or through fields such as $stateManager, $store, or state.

The binding is path-based. The model does not copy data and then try to sync later. It writes through to the bound source.

Examples

Standalone model

const user = new User({
    id: 1,
    name: 'Ankit',
    profile: { city: 'Jaipur' },
    roles: [
        { id: 1, name: 'Admin' },
        { id: 2, name: 'Editor' }
    ]
});

user.name = 'Ankit Karan';
user.profile.city = 'Kota';
user.roles.pushModel({ id: 3, name: 'Viewer' });

Standalone collection

const roles = Collection.from(Role, [
    { id: 1, name: 'Admin' },
    { id: 2, name: 'Editor' }
]);

roles.pushModel({ id: 3, name: 'Viewer' });
roles.updateById(2, { name: 'Content Editor' });
console.log(roles.findById(2)?.name);

Reset collection from plain array data

user.roles.resetModels([
    { id: 10, name: 'Owner' },
    { id: 11, name: 'Manager' }
]);

Iteration

for (const role of user.roles) {
    console.log(role.name);
}

const names = user.roles.pluck('name');

Caveats and Limitations

Important: hasMany items are index-path based in reactive mode. If you keep a saved reference like const role = user.roles.at(0) and later call unshift, splice, sort, or remove items, that old reference can become stale.
  • Property access can still collide with model method names. Example: a data field named clone will be shadowed by the method.
  • Collection numeric access like roles[0] is supported for reading, not for array-like assignment magic.
  • raw() expects object-shaped state for models. If the bound path is not an object, the model throws instead of pretending everything is fine.
  • Collection patch methods such as updateAt and updateBy require plain object patches.

Best Practices

  • Use findById() or fresh at() lookups after structural array changes instead of holding long-lived child references.
  • Use cloneWith() when you want detached immutable-style copies.
  • Use resetModels() when replacing a relation from plain data and you want model normalization.
  • Use peek() when you need raw source values without relation wrapping.
  • Keep relation data object-shaped for hasOne and array-shaped for hasMany.
